##开发中遇到的主要问题及解决方式

1.AngularJs中ng-repeat嵌套问题：

在这里需要使用track by表达式，因为这样的循环嵌套，使数组中出现重复的内容，数组中出现重复的内容，意味着它没有办法被唯一标示，而ng-repeat不允许出现这样的情况，为了避免这样的情况发生，需要我们定义track by表达式来区分不同的数组中的子元素。
感觉track by这里水还很深，不太敢轻易使用，因为在使用时遇到了一些问题—>在嵌套的ng-repeat中使用下拉菜单来切换状态时，如果用了track by \$index那么在切换状态时改变的只是ng-repeat中的数据，而结构上的状态并不会重新载入一遍。而且在多层嵌套时，也时常因为\$index互相冲突而产生问题。

###2.文件上传：
使用input type=file上传文件，文件上传与普通数据跟后台的交互不同，上传文件时需要获取更多的文件相关参数。
这里后来使用angular-file-update组件解决。
angular-file-update组件使用：需要先实例化一个FileUploader对象

这里用到了两个参数，url和autoUpload，url自然不用说，autoUpload值置为true，可以在触发uploader时自动发送文件。

###3.在使用setTimeout时，数据无法及时更新到view中：

####AngularJS中数据双向绑定——\$apply和\$digest

For example, if you use JavaScript’s setTimeout() function to update a scope model, Angular has no way of knowing what you might change. In this case it’s your responsibility to call \$apply() manually, which triggers a \$digest cycle. Similarly, if you have a directive that sets up a DOM event listener and changes some models inside the handler function, you need to call $apply() to ensure the changes take effect.

如果你使用了 JavaScript 中的 setTimeout() 来更新一个 scope model ，那么AngularJS 就没有办法知道你更改了什么。这种情况下，就需要手动调用\$apply()了，通过调用它来触发一轮 \$digest 循环。类似地，如果你有一个directive用来设置一个DOM 事件 listener 并且在该 listener 中修改了一些 models ，那么你也需要通过手动调用 \$apply() 来确保变更会被正确的反映到 view 中。

Note: By the way, you should use \$timeout service whenever possible which is setTimeout() with automatic \$apply() so that you don’t have to call \$apply() manually.

Note: 顺便提一下，你应该使用 \$timeout service 来代替 setTimeout() ，因为前者会帮你调用 \$apply() ，让你不需要手动地调用它。

Keep in mind that you should always use the version of \$apply() that accepts a function argument. This is because when you pass a function to \$apply(), the function call is wrapped inside a try…catch block, and any exceptions that occur will be passed to the \$exceptionHandler service.

需要记住的是你总是应该使用接受一个 function 作为参数的 \$apply() 方法。这是因为当你传入一个function 到 \$apply() 中的时候，这个 function 会被包装到一个 try … catch 块中，所以一旦有异常发生，该异常会被 $exceptionHandler service 处理。
